# AOS v3.1 系统清理与一致性修复

**日期:** 2026-03-05
**Agent:** God (Orchestrator)

## 🎯 目标
消除从 `dispatch-v2` / `ATO` 过渡到 **AOS v3.1** 产生的技术债务，并强制执行反映“内核-效应器-接口”架构的统一文件结构。

## 🛠️ 关键行动

### 1. 目录整合 (`skills/aos`)
将遗留的 `agent-orchestration-system` skill 合并到面向用户的 `aos` skill 中，建立单一的事实来源。
- **内核 (`core/`)**: `autopilot`, `dispatch_router`, `queue_sync`
- **效应器 (`effectors/`)**: `heartbeat_full`, `spawn_runner`, `execute_actions`
- **接口 (`cli/`)**: `status`, `sprint`, `review`

### 2. Cron 任务修复
发现活跃的 **AOS Heartbeat** cron 任务指向了已删除的路径 (`skills/agent-orchestration-system/...`)。
- **修复**: 更新 cron payload 指向新的 `skills/aos/scripts/core/` 和 `skills/aos/scripts/effectors/` 路径。
- **结果**: 防止了自动化流水线的静默失败。

### 3. Agent 身份对齐
发现子 Agent (COO, CTO, CMO, Reviewer) 的 `SOUL.md` 和 `IDENTITY.md` 文件在运行时目录中缺失。
- **行动**: 将权威提示词从仓库部署到 `agents/<role>/`。
- **影响**: 确保 Agent 正确体现其 AOS v3.1 角色（例如，“只读运行者”，“确定性合并”）。

## 🧠 反思
系统的弹性在此次重构中经受住了考验。在不重启的情况下“热修复”运行中的 cron 任务展示了 OpenClaw 架构的强大之处。未来，目录结构的变更必须立即同步到 `openclaw.json` 配置中。
